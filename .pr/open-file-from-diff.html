Open a note straight from the diff view

The history viewer's diff pane groups changes by file under a header like modified test.md. Those headers were inert text. This change adds an open-file button (↗) to the right of each header that opens the note in the workspace — threading an optional onOpenFile callback from the plugin down through the modal into the renderer, plus a new path-resolution helper that maps the diff's repo-root-relative path back to a vault TFile.

1What changed

★ The whole feature is one optional callback. When onOpenFile is supplied, each diff file header gains a clickable ↗ icon; clicking it hands the file's repo-relative path to the plugin, which resolves it to a real note and opens it. When the callback is absent, rendering is byte-for-byte as before — so nothing else in the diff view is affected. diff.ts:105-119

Three layers participate, and the only genuinely new logic is the path mapping in openDiffFile (§4). Everything else is plumbing the callback through to the place a click happens.

2Wiring: one callback, three layers

The click originates deep inside the pure renderer (renderDiffInto) but the ability to open a file lives on the plugin (it knows the vault + git repo root). So a callback is threaded down; the resolved path travels back up on click.

ⓘ Read the orange edge as the only new runtime flow: a click travels back up out of the otherwise-pure renderer. Blue edges are existing call structure now carrying one extra argument.

4Path resolution — src/main.ts

Sources src/main.ts — showHistory (repoRoot + callback), new openDiffFile

This is the real logic. Git diff paths are relative to the repo root, but Obsidian's getAbstractFileByPath wants a vault-relative path — and the vault may sit anywhere under the repo root. openDiffFile walks a three-tier resolution ladder, then opens in the current leaf or warns.

private async openDiffFile(repoRoot: string | null, repoRelPath: string) {
  const base = this.vaultBase();
  let file: TAbstractFile | null = null;
  // tier 1 — map repo-root-relative diff path back into the vault
  if (repoRoot) {
    const abs = `${repoRoot}/${repoRelPath}`;
    if (abs.startsWith(`${base}/`)) file = vault.getAbstractFileByPath(abs.slice(base.length + 1));
  }
  // tier 2 — vault IS the repo root, so the diff path is already vault-relative
  if (!(file instanceof TFile)) file = vault.getAbstractFileByPath(repoRelPath);
  if (file instanceof TFile) { await workspace.getLeaf(false).openFile(file); return; }
  // tier 3 — last resort: resolve by basename across the vault
  const resolved = metadataCache.getFirstLinkpathDest(repoRelPath.split("/").pop() ?? repoRelPath, "");
  if (resolved) { await workspace.getLeaf(false).openFile(resolved); return; }
  new Notice(`Git Lens: can't open "${repoRelPath}" — not in this vault`);
}

The caller resolves the repo root once per history open and curries it into the callback, so the per-click handler does no extra git work: main.ts:291 (repoRoot = await git.getRepoRoot(abs)) · :302 (p => openDiffFile(repoRoot, p)).

6Notes & edge cases

  • Deleted files still show the button, but won't resolve to a TFile — the user gets the "can't open" Notice rather than the button being hidden. Intentional; can be suppressed per-kind if desired.
  • Renames use file.path (the new path), so the button opens the current location.
  • Tier 3 uses getFirstLinkpathDest (basename match), which is also why a moved-but-same-name note still opens.
  • Verified: npm run build (tsc --noEmit + esbuild) passes clean. No existing tests touch this code path.
